刚开始理解 RAG 时，我很容易把它想成“把文档丢给大模型”。后来真正拆开链路才发现，RAG 的核心不是把资料塞进 prompt，而是让系统在回答之前，能从大量文档里找到更可能相关、可追溯、且有权限使用的片段。它更像一个检索工程，而不是一个简单的提示词技巧。

从资料里看到的知识库流程，大体可以分成两段：文档处理和问答处理。文档处理决定“知识怎样被系统理解”；问答处理决定“用户的问题怎样找到正确片段”。这两段任何一处做得粗糙，最后都会以模型幻觉的形式表现出来。模型可能说得很流畅，但它依据的上下文如果错了，流畅反而会更危险。

文档入库的第一步通常是文件上传。系统会有一个统一的文件上传接口，不同业务通过 `businessType` 区分，文件先落到文件总库，返回 `fileId`。这个设计看起来很普通，但很重要。因为知识库文档不是孤立文本，它需要知道来源、业务类型、所属知识库、上传人、权限范围和后续处理状态。没有这些元数据，后面的检索、追踪和权限控制都会变得模糊。

上传之后，后端会对文件做解析。这里常用 Apache Tika，因为它支持从多种文档格式中提取文本。解析结果可以看作一个 `Document`，但真正有价值的不是只有正文，还包括元数据。给 Document 添加标签、来源、文件 ID、业务类型、知识库 ID 等信息，本质上是在给后续召回增加可过滤、可解释的维度。RAG 不是只靠语义相似度，很多时候“这段内容来自哪里”同样影响它能不能被使用。

接下来是分块。文档为什么要分块？最直接的原因是 embedding 模型对输入长度有限制，文档太长无法一次向量化。第二个原因是语义匹配需要合适的颗粒度。RAG 通过语义相似度找片段，如果一个 chunk 太大，里面混入很多无关信息，匹配会变钝；如果切得太碎，一个完整事实又可能被拆散。第三个原因是大模型上下文窗口和 token 成本。最终进入 prompt 的内容越长，成本越高，干扰也越多。

分块没有绝对最优，更多是取舍。固定大小分块实现简单，比如按 300 或 512 token 切，必要时做 overlap。它便宜、稳定，但容易把一个语义单元切断。语义分块会先按段落或句子初步分段，再通过 embedding 判断相邻段落是否应该合并。它更能保持语义连贯，但需要设定相似度阈值，计算成本也更高。递归分段会按标题、换行、标点等分隔符逐层切分，直到块大小合适，效果介于固定分块和语义分块之间，但很依赖文档本身质量。文档结构分块适合格式清楚的材料，比如按标题层级、章节、表格说明来切，成本低，但遇到结构混乱的文档就容易失效。基于 LLM 的分段质量可能更好，但速度慢、成本高、模型选择也会影响稳定性。

所以分块不是一个纯算法问题，而是工程选择。你要在召回率、精度、成本、速度和维护复杂度之间找平衡。企业知识库里常见的文档并不总是干净的，有扫描件、表格、标题不规范、复制粘贴痕迹，也有一段话里混着制度、流程和例外情况。分块策略再聪明，也需要承认输入质量会限制输出质量。

问答处理从用户输入开始，但用户输入经常不是一个完整查询。资料里举的例子很典型：第一轮用户问“iPhone 14 多少钱？”，第二轮问“那有分期吗？”如果把第二句单独送去检索，系统只能看到“分期”，可能召回信用卡分期、电商分期规则等泛化内容，却丢了核心主语 iPhone 14。Query Rewriting 的价值就在这里：结合历史对话，把当前问题改写成“iPhone 14 有分期付款服务吗？”这样的完整查询。

我理解 QueryRewrite 不是润色，而是补齐检索所需的信息。它应该保留实体、补全省略、拆开复合意图，同时避免把用户没有表达的东西强行加进去。改写过度会引入新偏差，改写不足又无法解决多轮对话里的主语缺失。好的 QueryRewrite 应该让检索系统更清楚地知道“要找什么”，而不是让问题看起来更漂亮。

召回部分，我更认同混合检索。单独依赖 pgvector 语义检索，可以处理同义表达和语义相近的问题，但它对编号、专有名词、岗位名、配置项、产品型号不一定稳定。PostgreSQL 全文检索则可以通过 `tsvector` 建立倒排索引，用 `to_tsquery` 或类似方式构建查询表达式，再用 `@@` 判断是否命中，用 `ts_rank` 计算相关性。关键词检索没有语义检索那么“聪明”，但它在精确命中实体时很可靠。

倒排索引可以理解成关键词到文档列表的字典目录。正排索引是“文档里有哪些词”，查询时需要遍历；倒排索引是“这个词出现在哪些文档”，查询时能直接定位候选。BM25 则是一种经典关键词相关性评分思想，它会考虑词频、文档长度和逆文档频率。一个词在某段中出现越多，相关性可能越高；但长文档不能天然占便宜，所以要做长度惩罚；越稀缺的词，区分度越强，权重也更高。

在实际链路里，可以让 QueryRewrite 后的查询同时走两路：一路向量化后通过 pgvector 做相似度检索，召回一批语义候选；另一路通过 PostgreSQL 全文检索，基于 `tsvector`、`@@` 和 `ts_rank` 召回关键词候选。两路分数体系不同，不能简单相加，因此可以用 RRF，也就是 Reciprocal Rank Fusion，按排名融合。RRF 的想法很朴素：一个文档如果在多个榜单里排名都靠前，它就更值得被保留；如果只在某一路里靠前，也仍然有机会进入候选。

融合之后还可以做 rerank 精排。粗召回阶段更关注覆盖率，宁愿多召回一些可能相关的片段；rerank 阶段再让模型或 cross-encoder 对 query 和候选文档联合判断，重新排序。最后只选 TopK，例如 Top5，进入大模型上下文。这个 TopK 不只是为了省 token，更是为了减少噪声。上下文里无关内容越多，模型越容易被带偏。

完整问答还要结合记忆和提示词构建。短期记忆可以从当前会话中取最近几轮，并通过滑动窗口加摘要控制长度；长期记忆可以把用户主动录入的信息向量化后召回。最终 prompt 不是单一 RAG 片段，而是用户问题、短期记忆、长期记忆、检索片段和系统约束的组合。这里的难点是排序和边界：什么信息应该靠前，什么信息只能作为参考，什么信息不能进入上下文。

降低幻觉风险，也不是一句“请不要编造”就能解决。更可靠的方式是把证据链路做扎实：文档有来源，片段有元数据，查询经过改写但可观察，召回结果可以追踪，权限过滤在模型生成前完成，最终回答尽量基于 TopK 片段。模型仍然可能犯错，但至少系统能知道错误可能来自哪里。

RAG 给我的一个提醒是：AI 工程里很多问题都不是单点优化。分块太粗，召回会散；分块太细，事实会断；只做向量，专有名词可能丢；只做关键词，语义问题又召不回来；rerank 能提高精度，但会增加延迟和成本；记忆能让对话连续，也可能带来旧信息干扰。工程落地不是追求某个环节“最强”，而是让整条回答路径在成本、速度和可信度之间保持清醒。

所以我现在看 RAG，不再把它当成一个“接入知识库”的功能，而是把它当成一个把混乱文档转成可检索上下文的过程。它要求我们承认文档会脏、问题会省略、召回会误差、模型会幻觉，然后一层一层设计缓冲和校验。技术上看是解析、分块、向量、全文检索、RRF、rerank；更本质地看，是在回答之前，先认真处理“依据从哪里来”。
