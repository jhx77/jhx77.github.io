秒杀系统之所以难，不是因为“下单”这个动作本身复杂，而是因为所有人几乎在同一时间做同一件事。平时分散的请求，在几秒钟里集中打到同一个优惠券、同一份库存、同一张表、甚至同一行数据上。系统面对的不是普通流量，而是一种被压缩到极短时间里的混乱。

在智荐生活平台这个项目里，秒杀只是综合生活服务的一部分。平台还包含商家信息查询、优惠券活动、订单处理和智能客服。但秒杀链路很适合拿来复盘，因为它把后端系统里很多核心问题集中到一起：入口如何挡住无效请求，库存如何避免超卖，订单如何异步生成，缓存如何稳定，支付和关单如何处理并发，系统如何在压力下保持可恢复。

最直观的问题是，数据库不能直接扛全部请求。不是说数据库不重要，而是它不适合站在流量入口最前面。秒杀时，如果每个请求都去数据库查库存、扣库存、查用户是否下过单，再创建订单，连接池、行锁、事务等待都会被放大。数据库应该处理最终可靠状态，而不是承受所有瞬时判断。

因此入口层先使用 Redis 预减库存。用户请求进来后，先在 Redis 中判断活动和库存，再判断用户是否已经下过单。这里最关键的是 Lua 脚本。库存扣减和一人一单判断不能拆成多个命令分开执行，否则两个请求可能同时通过检查，再分别扣减，留下并发窗口。Lua 把这些逻辑放进同一个原子操作里：库存不足直接失败，用户已经下过单直接失败，通过校验才预扣库存并记录用户参与状态。

这一步解决的是入口秩序。它不代表订单已经真正落库，而是先把不该进入后续链路的请求挡掉。通过 Redis 校验后，请求会进入 Kafka，由消费者异步处理库存扣减和订单生成。同步链路只负责资格判断和预扣，真正写数据库的动作放到消息队列后面。Kafka 的意义是削峰，让数据库按照可承受的节奏处理订单，而不是被瞬时流量推着走。

异步并不等于简单。消息可能重复投递，消费者可能处理失败，数据库写入也可能超时。所以订单生成需要幂等。可以通过用户和优惠券的唯一约束兜底一人一单，也可以在消费逻辑里根据业务键判断消息是否已经处理。Redis 中的预扣库存、Kafka 中的订单消息、数据库里的最终订单状态，三者之间需要通过成功、失败、超时和补偿慢慢收敛。

缓存是另一个关键点。生活服务平台里，商家详情、优惠券信息、活动状态都可能成为热点数据。缓存问题通常会被分成三类：缓存击穿、缓存穿透和缓存雪崩。缓存击穿是某个热点 key 过期后，大量请求同时回源数据库；缓存穿透是请求查询根本不存在的数据，绕过缓存持续打到数据库；缓存雪崩则是大量 key 同时失效，导致数据库突然承压。

针对热点 key，逻辑过期是一种更稳的选择。缓存里不只存数据，还存一个逻辑过期时间。请求读到过期数据时，可以先返回旧值，再由一个线程拿锁去后台重建缓存。这样用户请求不用同时等待数据库，也避免大量请求一起回源。它的代价是短时间内可能读到旧数据，所以更适合能接受短暂延迟一致性的热点展示数据。

针对缓存穿透，可以缓存空值。比如查询一个不存在的商家 ID，如果每次都直接打数据库，恶意请求或错误参数会造成不必要压力。把空结果短时间缓存起来，可以让后续相同请求直接返回。这里也要控制 TTL，避免空值长期占据缓存，影响正常数据写入后的可见性。

多级缓存则是为了进一步减少访问路径。Caffeine 本地缓存放在应用实例内，速度快，适合承接短时间重复读；Redis 作为分布式缓存，负责多实例共享；数据库作为最终数据源。Caffeine + Redis 的二级缓存结构并不是越多越好，而是要清楚每一层承担什么：本地缓存减轻 Redis 压力，Redis 减轻数据库压力，数据库保证最终状态。

订单状态比库存更容易被忽略。用户点击支付、支付平台回调、定时任务关闭订单，这些动作可能在时间上交错。比如订单刚好超时，Spring Task 正准备关闭；与此同时支付回调到达，系统要判断谁能更新成功。这里需要乐观锁或状态条件更新：只有当前状态仍是待支付时，才能改成已支付或已关闭。已经关闭的订单不能被迟到回调重新改成支付成功，已经支付的订单也不能再被定时任务关闭。

未支付订单自动关闭可以用 Spring Task 定时任务处理。它扫描超过支付期限仍未支付的订单，执行关闭并触发必要的库存补偿。定时任务不是为了炫技，而是承认分布式系统里一定会有请求失败、消息延迟和用户中途离开。同步接口不能保证所有状态都立刻完美结束，后台补偿就成了让状态回到合理位置的一部分。

限流也很重要。秒杀活动天然会吸引重复点击、刷券和爬虫请求。使用 Redis + AOP + 注解实现滑动窗口限流，可以把限流规则和业务代码解耦。注解定义限流维度和阈值，AOP 在接口前统一拦截，Redis 记录窗口内访问次数。维度可以是全局、IP、用户等。全局限流保护系统整体，IP 限流减少异常来源压力，用户维度限制重复行为。

这里的“滑动窗口”比固定窗口更细一些。固定窗口可能在窗口切换瞬间被打满两次，而滑动窗口会根据当前时间向前统计一段范围内的请求，更接近真实频率。当然，它也会带来更多 Redis 操作和数据清理成本。限流从来不是免费的，它是在可用性和公平性之间做约束。

项目里还有 AI 智能客服部分。它接入阿里云百炼大平台模型，使用 LangChain4j 做大模型调用，Redis 存储会话上下文，并通过 Function Calling 实现信息查询、预约等业务逻辑。这个部分和秒杀看起来不在同一层，但有一个共同点：模型回答不能脱离真实业务状态。客服如果要查订单、查商家、查预约，就应该通过受控函数获取信息，而不是让模型凭语言猜。

回头看这个项目，我觉得高并发系统的本质不是把请求全部“接住”，而是让混乱变得有序。Redis 在入口做快速判断，Lua 保证原子性，Kafka 把峰值削开，数据库保存最终状态，缓存承接热点读，定时任务处理迟到的问题，限流承认系统有边界。每个组件都不是为了让架构看起来复杂，而是在某个压力点前放一个缓冲。

这和生活里的节奏管理有一点相似。一个人也不可能永远无限响应外界请求。缓存像经验，能让我们不必每次从零开始；限流像边界，提醒我们不能什么都接；削峰像安排优先级，把一瞬间涌来的事情摊开处理；补偿机制像复盘，承认有些事当下处理不好，之后还要把状态收回来。

当然，技术不是人生鸡汤。系统设计最终还是要回到具体问题：哪里会并发，哪里会失败，哪里需要原子性，哪里可以异步，哪里必须兜底。稳定不是假装问题不会出现，而是提前承认它们会出现，然后设计缓冲、隔离和恢复。秒杀系统给我最大的启发，就是秩序不是静态的，它是在压力、失败和约束之间被一点点设计出来的。
